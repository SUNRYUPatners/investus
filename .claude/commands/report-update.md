# /report-update — 날짜별 리포트 전체 업데이트 세트

새 날짜의 리포트를 추가할 때 반드시 아래 순서를 **전부** 실행한다. 하나라도 빠지면 미완성이다.

## 실행 순서

### 1. SVG 생성
- `public/charts/` 폴더에 해당 날짜 SVG 파일 생성
- **한국어판 + 영어판(-en suffix) 모두 필수** — 모든 차트에 KO/EN 쌍 생성
- 파일명 규칙: `<종목>-<설명>-<YYYYMMDD>.svg` (영어판: `<종목>-<설명>-<YYYYMMDD>-en.svg`)
- **viewBox 높이 계산 철저히** — 요소 겹침 절대 금지

#### SVG 스타일 — 2026-07-15 업데이트: 세부 규칙 폐기, 실제 파일을 레퍼런스로

> **세부 폰트/박스 규칙을 글로 규정하지 않는다. 대신 아래 3개 날짜의 실제 SVG 파일을 매번 Read로 열어보고 그 구조를 그대로 따라 만든다.**
>
> 레퍼런스 (새 SVG 만들기 전에 비슷한 소재의 파일을 1~2개 Read):
> - `public/charts/*20260702*.svg`
> - `public/charts/*20260703*.svg`
> - `public/charts/*20260704*.svg`

- 3개 날짜 스타일이 서로 다르다 (색 테마, 배경 그리드, row 구성) — 오늘 소재 성격에 맞는 걸 골라 구조를 베낀다
- 제목·히어로 숫자는 크게, 카드형 KPI 블록으로 densely 채우기 — 빈 공백 남기지 않기
- **"투자 시사점" / "Investment Implications" 박스를 이미지에 넣는 것도 허용** (7/4 `tsla-fsd-manslaughter-20260704.svg` 참고) — 단 `lib/reports.ts`의 body/bodyEn에는 항상 별도로 채워야 함
- 상세 배경 전체는 그래도 `lib/reports.ts` body에 충분히
- 색상: TSLA=`#00e5a0`/`#4ade80`, SPCX=`#c084fc`, NVDA/TSM/META=`#60a5fa`, 위험/BREAKING=`#ef4444`/`#f97316`
- XML에서 `&` → `&amp;` 필수
- 생성 후 sharp로 PNG 렌더링해서 실제로 눈으로 잘림/오버플로 확인

#### 한장요약 SVG

- 레퍼런스: `summary-20260702.svg`, `summary-20260703.svg`, `summary-20260704.svg` — densely 채운 그리드/카드 형태
- 제목에 티커를 잔뜩 나열하지 말고 “핵심 N가지” 형태 권장

### 2. lib/reports.ts — seed 추가
- `SEED_REPORTS` 배열에 새 seed 삽입 (pinned 요약 카드가 가장 앞)
- `REPORT_TICKERS` 맵에 seed별 ticker 배열 추가
- 날짜별 섹션 주석으로 구분
- **본문에 글 충분히 채울 것** — SVG에 못 넣은 배경·스펙·출처 상세는 body에
- **개별 리포트마다 `■ 투자시사점` / EN `■ Investment Implications` 필수** (사실만 나열하고 끝내지 말 것)

### 3. lib/wallPosts.ts — 종토방 글/댓글 추가
- 날짜 타임스탬프 상수 추가 (`const T<DD>J = _now - <n> * 3600_000`)
- `MOCK_POSTS` 배열에 새 날짜 게시글 추가 (ID는 기존 최대값+1부터)
- `MOCK_COMMENTS` 맵에 해당 게시글 댓글 추가
- `LATEST_UPDATE`를 오늘 날짜 타임스탬프로 갱신

### 4. lib/analystPosts.ts — 애널리스트 글/댓글 추가
- `MOCK_ANALYST_POSTS` 배열에 새 날짜 포스트 추가 (id는 음수, 최소값-1부터)
- `MOCK_ANALYST_COMMENTS` 맵에 댓글 추가
- **당일 개별 리포트마다 1개씩 커버 필수** — 개별 리포트 수 = 애널 포스트 최소 수. 빠진 리포트 없이 전부 커버.

### 5. scripts/export-report-pngs.js — PAIRS 추가
- `PAIRS` 배열에 오늘 날짜 항목 추가
- `{ ko: '파일명.svg', en: 'en파일명.svg' or null, label: '라벨', date: 'YYYYMMDD' }` 형식
- 영어판 없으면 `en: null`

### 7. 배포 전 팩트체크 (필수 — 생략 불가)

> ⛔ **이 단계 없이 commit/deploy 절대 금지**
> ⛔ **"팩트체크 완료" 선언은 실제로 스크린샷을 다시 Read한 후에만 가능**
> ⛔ **기억·추론으로 대체 절대 금지 — 반드시 원본 스크린샷 파일을 다시 열어서 눈으로 확인**

#### 팩트체크 의무 절차 (순서대로 전부 실행)

**Step 1 — 핵심 수치 목록 추출**
작성한 모든 리포트에서 아래 항목을 표로 정리한다:
| 리포트명 | 수치/가격/% | 출처 스크린샷 파일명 |
|---------|------------|-------------------|
| (예) SPCX RJ PT | $800 | 스크린샷 7:37:36.png |

**Step 2 — 스크린샷 원본 재독 (Read tool 사용 필수)**
위 표의 각 항목마다 해당 스크린샷 파일을 `Read` tool로 **실제로 다시 열어서** 숫자를 직접 눈으로 확인한다.
- 기억에 의존하거나 "아까 봤으니까 맞겠지" 하는 추론 절대 금지
- 스크린샷 글씨가 작아서 불명확하면 불명확하다고 명시하고 가장 보수적인 숫자 사용

**Step 3 — 1:1 대조 체크리스트**
- [ ] 가격·목표주가 → 스크린샷 원문 숫자와 정확히 일치
- [ ] 퍼센트(%) → 방향(+/-)과 수치 모두 일치
- [ ] 날짜 → 원문 날짜와 일치
- [ ] 회사명·종목·사람 이름 → 오탈자 없음
- [ ] 방향성(상승/하락, 매수/매도) → 원문과 동일
- [ ] 없는 사실 창작 여부 → 스크린샷에 없는 내용 0건

**Step 4 — 불일치 발견 시**
즉시 수정 → 해당 SVG·ts 파일 재생성 → 스크린샷 재확인 → 체크리스트 재통과

**⚠️ 과거 오류 사례 (반복 금지)**
- 2026-07-10: SPCX Raymond James PT를 스크린샷 "$800"을 "$8,800"으로 잘못 기록
  → 원인: 스크린샷을 다시 Read하지 않고 기억으로 판단
  → 결과: 배포 후 수정 → 불필요한 토큰·시간 낭비

불일치 0건 확인 후에만 다음 단계 진행.

## 중요 규칙
- 1 리포트 = 1 주제 — 하나의 seed에 여러 주제 섞지 않는다
- **같은 종목이라도 이벤트/주제가 다르면 반드시 별도 리포트로 분리** — 한 seed에 독립적인 여러 이벤트 묶지 않는다 (예: 한국 FSD 구독 개시와 스페인 FSD 승인은 별도 seed)
- 날짜별 타임스탬프는 상대 오프셋으로 계산 (`_now - N*3600_000`)
- 배포 전 TypeScript 오류 0개 확인 필수
- **PNG 내보내기는 오늘 새로 만든 SVG만** — 이전 날짜 이미지 재생성 절대 금지
- **SVG는 `*20260702*.svg`/`*20260703*.svg`/`*20260704*.svg`를 직접 Read해서 그대로 따라 만든다** / 본문에는 항상 상세 글 + 투자시사점 (스킬: `.cursor/skills/investus-report-update/`)

## 종토방(wallPosts.ts) 글 작성 규칙
- 닉네임은 `익명_XXXX` 형식으로 4자리 숫자 무작위하게 배정
- 말투는 구어체 자연스럽게, 분석+개인 경험+감상 섞기
- 같은 날 같은 ticker 게시글이라도 논점/각도가 서로 달라야 한다
- holdingLabel은 "관심종목", "XX주 보유" 등 다양하게 배분

## 애널리스트(analystPosts.ts) 글 작성 규칙 — 절대 금지 패턴
- **금지**: 같은 날 같은 alias 두 번 이상 사용 금지 (예: "여의도 매 #17"을 같은 날 반복 불가)
- **금지**: 모든 글을 "~정리임", "~세트임"으로 시작하거나 끝내는 패턴 반복 금지
- **금지**: 모든 글이 "X종 세트임", "Y가지 정리" 식의 같은 구조로 시작하는 것 금지
- **다양한 오프닝 스타일** (날짜별로 섞어서 사용):
  - 채널 체크 뷰: "~쪽 채널 체크해봤는데", "~딜 플로우 확인했는데"
  - 숫자/데이터 포인트: "~보고 한 가지 숫자가 눈에 들어왔어"
  - 심층 분석: "~발언 다시 뜯어봤는데"
  - 검증: "~결과랑 딱 맞아 떨어져서"
  - 현장감: "~사진 봤는데, 속도가 예상보다"
  - 단도직입: "~가 오늘 핵심이야. 이유는..."
- 글마다 다른 관점(강세/중립/분석/의문) 배분 — 전부 강세로 쓰지 않는다
- alias는 ALIAS_POOL 전체에서 고르게 배분 (특정 alias 반복 지양)
